> **版本**: 1.0
> **创建日期**: 2025-12-17
> **状态**: 设计审阅中

---

## 1. 设计目标

### 1.1 核心目标

- **简洁直观**: 界面一目了然，降低使用门槛，用户无需学习即可上手
- **高效操作**: 常用功能触手可及，减少点击次数
- **实时反馈**: 监控状态、变化通知通过 SSE 即时推送到前端
- **响应式设计**: 支持桌面和移动端访问，自适应不同屏幕尺寸

### 1.2 设计原则

- **高内聚、低耦合**: 前后端模块独立，职责清晰，便于维护和扩展
- **渐进式复杂度**: 基础功能简单易用，高级功能按需展开
- **数据驱动**: UI 状态与后端数据实时同步，避免手动刷新
- **容器化部署**: 支持 Docker 打包，便于一键部署

## 2. 技术选型

### 2.1 前端技术栈

| 类别 | 技术选型 | 说明 |
|------|----------|------|
| 框架 | Vue 3 + TypeScript | Composition API 更适合复杂状态管理 |
| 构建 | Vite | 构建速度快，开发体验好 |
| UI 库 | Naive UI | 设计现代、组件丰富、支持暗色主题、Vue 3 原生支持 |
| 状态管理 | Pinia | Vue 3 官方推荐，轻量灵活 |
| 路由 | Vue Router 4 | SPA 路由管理 |
| HTTP | Axios | 请求封装 |
| 图表 | ECharts (可选) | 用于统计图表展示 |
| 样式 | UnoCSS | 原子化 CSS，按需生成 |

### 2.2 后端技术栈

| 类别 | 技术选型 | 说明 |
|------|----------|------|
| 框架 | FastAPI | 与现有 Python 代码无缝集成，原生异步支持 |
| 异步 | asyncio + uvicorn | 高性能异步服务器 |
| 实时通信 | SSE | 比 WebSocket 更轻量，单向推送场景更合适，穿透代理更容易 |
| 文档 | OpenAPI/Swagger | 自动生成 API 文档，便于前后端协作 |

### 2.3 SSE vs WebSocket 选型理由

选择 SSE (Server-Sent Events) 作为实时通信方案：

1. **协议简单**: 基于 HTTP，无需额外协议握手，穿透代理和防火墙更容易
2. **自动重连**: 浏览器原生支持断线自动重连，无需额外实现
3. **单向推送**: 监控场景主要是服务端向客户端推送，SSE 更匹配
4. **RESTful 风格**: 客户端操作通过 POST 请求完成，保持 API 风格统一
5. **实现简单**: 服务端只需返回 StreamingResponse，无需维护连接状态

## 3. 系统架构

### 3.1 分层架构

系统采用四层架构设计：

```
┌─────────────────────────────────────────────────────────────────┐
│                         WebUI Layer                              │
│        Dashboard  |  Tasks  |  History  |  Settings              │
├─────────────────────────────────────────────────────────────────┤
│                       API Gateway (FastAPI)                      │
│          REST API  |  SSE  |  Auth  |  Static Files              │
├─────────────────────────────────────────────────────────────────┤
│                      Core Service Layer                          │
│     (复用现有模块: TaskManager, Scheduler, Notification, etc.)   │
├─────────────────────────────────────────────────────────────────┤
│                       Storage Layer                              │
│              config.json  |  history.json  |  Logs               │
└─────────────────────────────────────────────────────────────────┘
```

**层次说明**:
- **WebUI Layer**: 前端 Vue 3 应用，负责用户交互和数据展示
- **API Gateway**: FastAPI 提供 RESTful API 和 SSE 事件流
- **Core Service Layer**: 复用现有业务模块，无需重复开发
- **Storage Layer**: 使用现有的 JSON 文件存储

### 3.2 目录结构

**后端模块** (`webmon/web/`):
- `app.py` - FastAPI 应用入口
- `api/` - API 路由模块 (tasks, history, settings, monitor, sse)
- `schemas/` - Pydantic 请求/响应数据模型
- `services/` - 业务逻辑封装，桥接 Core Layer
- `static/` - 前端构建产物

**前端模块** (`frontend/`):
- `views/` - 页面组件 (Dashboard, Tasks, History, Settings)
- `components/` - 通用 UI 组件
- `stores/` - Pinia 状态管理
- `api/` - API 调用封装
- `types/` - TypeScript 类型定义

## 5. API 设计

### 5.1 RESTful API

采用标准 RESTful 风格，所有接口以 `/api` 为前缀。

**任务管理**:
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/tasks | 获取任务列表 |
| POST | /api/tasks | 创建任务 |
| GET | /api/tasks/{id} | 获取任务详情 |
| PUT | /api/tasks/{id} | 更新任务 |
| DELETE | /api/tasks/{id} | 删除任务 |
| POST | /api/tasks/{id}/check | 立即检测任务 |
| PATCH | /api/tasks/{id}/enable | 启用任务 |
| PATCH | /api/tasks/{id}/disable | 禁用任务 |

**监控控制**:
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/monitor/status | 获取监控状态 |
| POST | /api/monitor/start | 启动监控 |
| POST | /api/monitor/stop | 停止监控 |

**历史记录**:
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/history | 获取历史记录 (支持分页、筛选) |
| GET | /api/history/{id} | 获取历史详情 |
| DELETE | /api/history/{id} | 删除历史记录 |
| DELETE | /api/history | 清空历史记录 |

**配置管理**:
| 方法 | 路径 | 说明 |
|------|------|------|
| GET | /api/settings | 获取所有配置 |
| PUT | /api/settings | 更新配置 |
| GET | /api/settings/{section} | 获取指定配置段 |
| PUT | /api/settings/{section} | 更新指定配置段 |

**其他**:
| 方法 | 路径 | 说明 |
|------|------|------|
| POST | /api/notification/test | 发送测试通知 |
| GET | /api/stats/overview | 获取概览统计 |

### 5.2 SSE 事件流

**端点**: `GET /api/events`

客户端通过 EventSource 建立连接，服务端持续推送事件。

**事件类型**:
| 事件名 | 说明 | 数据内容 |
|--------|------|----------|
| status_update | 监控状态更新 | status, active_tasks |
| task_checked | 任务检测完成 | task_id, success, changed |
| change_detected | 检测到变化 | task_id, task_name, timestamp, changes_count, ai_summary |
| error | 错误通知 | task_id, message |
| notification_sent | 通知已发送 | platform, success |
| heartbeat | 心跳保活 | time (每30秒) |

**特点**:
- 浏览器原生支持自动重连
- 服务端通过 StreamingResponse 实现
- 心跳机制保持连接活跃

### 5.3 数据模型

任务相关模型包含以下字段：
- 创建任务: url*, name*, description, selectors, interval, timeout, enabled, ai_prompt
- 更新任务: 所有字段可选
- 响应任务: 包含 id、时间戳、统计信息、错误信息等完整字段

## 6. 状态管理

### 6.1 Store 划分

采用 Pinia 进行状态管理，按业务领域划分 Store：

**taskStore**: 管理任务数据
- 状态: tasks 列表、loading 状态、error 信息
- Getters: 活跃任务、按 ID 查找
- Actions: CRUD 操作、立即检测

**monitorStore**: 管理监控状态
- 状态: 运行状态、最后更新时间、最近变化列表
- Actions: 启动/停止监控、添加变化记录

**settingsStore**: 管理配置数据
- 状态: notification、ai、monitoring 等配置段
- Actions: 获取/更新配置

### 6.2 SSE 集成

通过 composable 函数 `useSSE()` 封装 EventSource 逻辑：
- 连接时注册各事件监听器
- 收到事件后更新对应 Store
- 自动处理重连 (浏览器原生支持)

## 9. 部署方案

### 9.1 开发模式

前后端分离开发：
- 后端: uvicorn 启动 FastAPI，支持热重载
- 前端: Vite 开发服务器，代理 API 请求到后端

### 9.2 生产模式

**方式一：直接运行**

前端构建后嵌入后端：
- 构建前端产物到 `webmon/web/static/`
- FastAPI 同时提供 API 和静态文件服务
- 单进程部署，简化运维

**方式二：Docker 部署**

提供 Dockerfile 和 docker-compose.yml，支持一键部署：

Dockerfile 构建流程：
1. 多阶段构建，第一阶段构建前端
2. 第二阶段基于 Python 镜像，安装依赖
3. 复制前端构建产物和后端代码
4. 暴露服务端口，设置启动命令

docker-compose.yml 编排：
- webmon 服务：主应用容器，挂载配置和数据目录
- 支持环境变量配置（端口、日志级别等）
- 数据持久化：挂载 config/、data/、logs/ 目录
- 可选：添加 Playwright 浏览器依赖镜像

部署命令：
- `docker-compose up -d` - 启动服务
- `docker-compose logs -f` - 查看日志
- `docker-compose down` - 停止服务

### 9.3 CLI 集成

新增 `web` 子命令：
- `python webmon.py web` - 启动 WebUI (默认 8000 端口)
- `python webmon.py web --port 8080` - 指定端口
- `python webmon.py web --no-browser` - 不自动打开浏览器
